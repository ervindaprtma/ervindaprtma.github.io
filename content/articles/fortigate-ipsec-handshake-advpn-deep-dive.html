FortiGate IPsec Deep Dive: Packet Handshake, Debug Flow, Hub-Spoke & ADVPN 2.0

  Ervinda Pratama   July 2026   ~22 min read

Introduction

Understanding IPsec only from GUI configuration is not enough for production engineers. Real operational mastery comes from understanding what happens at packet level during tunnel establishment, route learning, and traffic forwarding.

This article breaks down FortiGate IPsec behavior in three common enterprise scenarios:

  1. Static peer IPsec between two firewalls
  2. Hub-and-Spoke IPsec (without ADVPN)
  3. Hub-and-Spoke IPsec with ADVPN / ADVPN 2.0 shortcut tunnels

This article also covers packet handshake stages, debug commands, and SD-WAN transport-group behavior in ADVPN 2.0.

Core IPsec Handshake Sequence

Regardless of topology, most FortiGate IPsec sessions follow this sequence:

1. Interesting traffic detected OR always-up tunnel starts
2. IKE Phase 1 negotiation begins
3. DH key exchange performed
4. Peer authentication (PSK / certificate)
5. IKE SA established
6. Phase 2 selectors negotiated
7. IPsec SA established
8. Encrypted ESP traffic starts

Useful FortiGate Debug Commands

diagnose vpn ike log-filter clear
diagnose debug reset
diagnose debug application ike -1
diagnose debug enable

get vpn ipsec tunnel summary
diagnose vpn tunnel list

For dataplane policy/session checks:

diagnose debug flow filter addr x.x.x.x
diagnose debug flow trace start 50

Disable debug after testing.

Scenario 1: Static Peer IPsec Between Two Firewalls

Topology

LAN-A --- FGT-A === Internet === FGT-B --- LAN-B

Packet-Level Handshake

FGT-A -> UDP 500 -> FGT-B   (IKE_SA_INIT / proposal)
FGT-B -> UDP 500 -> FGT-A   (proposal accepted)
DH exchange occurs
FGT-A <-> FGT-B authenticate PSK/cert
IKE SA established
Phase2 Quick Mode / CHILD SA negotiated
ESP (Proto 50) traffic begins

Debug View (Typical)

initiating Main Mode / IKEv2 exchange
proposal chosen
peer authenticated
IPsec SA established

Traffic Flow After Tunnel Up

HostA 10.1.0.10 -> HostB 10.2.0.10
FGT-A encrypts -> Internet -> FGT-B decrypts -> LAN-B

Scenario 2: Hub-and-Spoke IPsec (Without ADVPN)

Topology

Spoke1 === Hub === Spoke2

Each spoke only forms tunnel to Hub.

Initial Routing Behavior

  • Spoke1 learns Spoke2 LAN via Hub.
  • Spoke2 learns Spoke1 LAN via Hub.
  • Traffic always traverses Hub.

Packet Path

Host-S1 -> FGT-Spoke1 -> IPsec to Hub
Hub decrypts / re-routes
Hub re-encrypts to Spoke2
FGT-Spoke2 decrypts -> destination

Operational Impact

TopicResult
LatencyHigher
Hub CPUIncreased
SimplicityHigh
East-West ScaleLimited

Debug Flow Example at Hub

Received ESP from Spoke1
Decrypt packet
Route lookup destination Spoke2 subnet
Match policy Hub->Spoke2
Encrypt and forward ESP to Spoke2

Scenario 3: Hub-and-Spoke With ADVPN

Topology

        Hub
       /   \
   Spoke1  Spoke2
      \     /
   Dynamic Shortcut Tunnel

ADVPN allows spokes to dynamically establish direct spoke-to-spoke shortcuts after initial discovery through the hub.

Initial Session Behavior

First packets often traverse Hub, then shortcut creation begins.

Packet Sequence

1. Spoke1 sends traffic to Spoke2 via Hub
2. Hub helps discovery / NHS-like broker role in Fortinet design
3. Spokes learn reachable peer underlay info
4. Shortcut IKE negotiation starts directly
5. Direct IPsec tunnel established
6. Subsequent traffic uses shortcut tunnel

Result

  • Lower latency
  • Lower hub load
  • Better east-west branch performance

ADVPN 2.0 Enhancements

Fortinet documents that ADVPN 2.0 introduced edge discovery and path management, with shortcut decisions made by spokes based on available underlays and health criteria. It reduces dependence on older routing design constraints such as route reflection or per-overlay BGP models. (docs2.fortinet.com)

ADVPN 2.0 also supports multiple shortcuts for load-balancing decisions depending on health and transport grouping. (docs.fortinet.com)

Transport Groups (ADVPN 2.0)

Transport groups operate under the following model:

Transport groups classify overlay members that belong to the same interconnected transport type (for example Internet-to-Internet, MPLS-to-MPLS). Shortcuts are not attempted between segregated transports assigned to different groups. (docs.fortinet.com)

Example

Overlay MemberTransportGroup
WAN1 BroadbandInternet1
WAN2 LTEInternet1
MPLS1MPLS2

Meaning

  • Spoke broadband can shortcut to remote LTE (both internet group).
  • Broadband to MPLS shortcut is normally not selected when groups differ.

Example Config

config system sdwan
 config members
  edit 1
   set interface "HUB_INET"
   set transport-group 1
  next
  edit 2
   set interface "HUB_MPLS"
   set transport-group 2
  next
 end
end

Debug Flow: ADVPN Shortcut Creation

Typical observable sequence:

Traffic enters hub path
Spoke evaluates remote spoke reachability
Shortcut query / discovery triggered
IKE negotiation direct between spokes
New child SA installed
Routing/session migrates to shortcut path

Important Troubleshooting Checks

  • PSK mismatch
  • Wrong selectors
  • BGP/OSPF routes missing across overlay
  • SD-WAN rule preference incorrect
  • NAT-T blocked UDP 4500
  • Shortcut disabled on phase1
  • Transport-group mismatch expectation
  • Session pinned to old path until re-evaluation

diagnose vpn ike gateway list

diagnose vpn tunnel list

Summary / Takeaways

  • Static peer IPsec creates direct permanent tunnels between known peers.
  • Traditional hub-and-spoke sends spoke-to-spoke traffic through the hub.
  • ADVPN dynamically builds direct shortcut tunnels after discovery.
  • ADVPN 2.0 improves spoke intelligence, path management, and transport-aware shortcut decisions.
  • Transport groups help ensure shortcuts use compatible underlay transports.
  • Debugging FortiGate IPsec requires understanding both control-plane (IKE) and data-plane (flow/session) behavior.

Effective VPN operations require understanding not only tunnel status, but also why the tunnel formed and how traffic is forwarded across it.
