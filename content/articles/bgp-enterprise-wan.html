Understanding BGP for Enterprise WAN Design

  Ervinda Pratama   March 2025   ~12 min read

Introduction

Border Gateway Protocol (BGP) is widely recognized as the routing protocol of the internet, but its value within enterprise WAN architecture is equally significant. In enterprise environments, BGP is used not only to exchange routes, but to enforce routing policy, control path selection, and preserve business continuity.

In multi-homed environments that depend on two or more ISPs, BGP becomes the decision engine that determines how traffic exits the organization and how external users reach internal services.

Redundant circuits without intelligent routing provide limited value. BGP transforms redundancy into operational resilience.

Why Enterprises Use BGP

Many organizations initially rely on static routes or default routes from a single ISP. This works in simple environments, but becomes limited when businesses require:

  • Dual or multi-ISP internet connectivity
  • Automatic failover during outage
  • Traffic engineering for critical applications
  • Public IP prefix advertisement
  • Better SLA and provider independence
  • Regional WAN edge optimization

iBGP vs eBGP: Internal and External Roles

Understanding Internal BGP (iBGP) and External BGP (eBGP) is fundamental.

Type Runs Between Main Purpose
eBGP Different Autonomous Systems Exchange routes with ISP / partners
iBGP Same Autonomous System Distribute learned routes internally

eBGP Characteristics

  • Used between enterprise edge router and ISP.
  • Default TTL assumes directly connected neighbors.
  • Typically changes NEXT_HOP to itself.
  • Administrative distance often preferred over many IGP external options.

iBGP Characteristics

  • Used between routers inside the same AS.
  • Does not change NEXT_HOP by default.
  • Requires full mesh or Route Reflector design.
  • Prevents re-advertising iBGP-learned routes to another iBGP peer unless reflection/confederation exists.

How BGP Chooses Best Path

BGP is a path-vector protocol. It selects the best route based on attributes rather than physical shortest path.

Common simplified decision sequence:

  1. Weight (Cisco proprietary)
  2. Local Preference
  3. Locally originated route
  4. Shortest AS Path
  5. Lowest Origin type
  6. Lowest MED
  7. eBGP over iBGP
  8. Lowest IGP metric to NEXT_HOP
  9. Oldest path / tie breakers

BGP prioritizes routing policy over physical distance.

Control Egress Traffic with Local Preference

When users inside the company browse the internet, this is outbound / egress traffic.

Use Local Preference to tell all routers which ISP should be preferred.

Higher value wins.

Scenario: Prefer ISP-A for all outbound traffic

router bgp 65001
 neighbor 1.1.1.1 remote-as 100
 neighbor 1.1.1.1 route-map PREFER-ISP-A in

route-map PREFER-ISP-A permit 10
 set local-preference 200

Result:

  • Routes learned from ISP-A become preferred internally.
  • Users exit through ISP-A unless unavailable.

Influence Inbound Traffic with AS-Path Prepending

Inbound traffic means how the internet reaches your company public services.

Because external networks make their own decisions, inbound control is indirect.

One common method is AS-Path Prepending.

You intentionally make one route look longer so remote networks prefer another provider.

Scenario: Make ISP-B Less Attractive

route-map PREPEND-ISP-B permit 10
 set as-path prepend 65001 65001 65001

Result:

  • Many networks prefer ISP-A path.
  • ISP-B remains backup or secondary entry path.

MED (Multi Exit Discriminator)

MED can suggest which entry point another AS should use when multiple links exist to the same provider.

Lower MED is preferred.

Use MED mainly when both circuits connect to the same ISP.

Designing High Availability WAN with BGP

True WAN resilience requires more than dual circuits.

Failure Scenario BGP Response
ISP-A outage Shift traffic to ISP-B
Fiber cut Withdraw prefixes / reroute
Edge router failure Secondary edge advertises path
Upstream instability Route reconvergence

High availability is achieved not by the presence of multiple links alone, but by intelligent failover and controlled path selection.

Recommendation Why

Use Route Reflector for scale Avoid iBGP full mesh complexity
Document policy changes Prevent asymmetric routing surprises

Common Mistakes in Enterprise BGP

  • Using BGP when static default route is enough for tiny branch sites.
  • No route filtering from ISP.
  • Ignoring asymmetric traffic flow.
  • Forgetting NEXT_HOP reachability in iBGP.
  • No graceful failover testing.
  • Advertising more prefixes than intended.

Summary / Takeaways

  • BGP is the strategic routing protocol for enterprise WAN multi-homing.
  • Use Local Preference to control how traffic leaves your network.
  • Use AS-Path Prepending or MED to influence how traffic enters.
  • iBGP distributes routes internally; eBGP connects to external AS networks.
  • Use Route Reflectors instead of full mesh at scale.
  • BGP enables resilient enterprise connectivity when internet availability is business critical.
