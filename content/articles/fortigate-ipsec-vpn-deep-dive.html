FortiGate IPsec Deep Dive: Encryption, DH Groups, Tunnel Design & Real Configuration

  Ervinda Pratama   June 2026   ~18 min read

Introduction

When organizations need to securely connect branches, data centers, cloud workloads, or remote partners across untrusted networks such as the internet, one of the most proven technologies is IPsec.

On FortiGate, IPsec is a core capability used for site-to-site tunnels, hub-and-spoke WAN, SD-WAN overlays, and secure hybrid-cloud connectivity.

IPsec does not change the trust model of the internet; it protects data confidentiality across untrusted networks.

Why Use IPsec?

IPsec is used when traffic must traverse public or shared networks securely.

What Is IPsec?

IPsec (Internet Protocol Security) is a suite of protocols that provides:

  • Confidentiality (encryption)
  • Integrity (anti-tamper validation)
  • Authentication (peer identity verification)
  • Anti-replay protection
  • Secure key exchange

It operates at Layer 3, protecting IP traffic.

What Is Diffie-Hellman?

Diffie-Hellman (DH) is a secure key exchange method allowing two peers to derive a shared secret over an untrusted network without sending the secret directly.

That shared secret is then used to derive encryption keys.

Without DH, secure key exchange over the internet would be difficult.

Practical Guidance

  • Group 14 = widely compatible baseline
  • Group 19/20 = modern preferred balance
  • Group 20 = excellent enterprise choice
  • Group 21+ = stronger but more CPU cost

What Is PFS?

Perfect Forward Secrecy (PFS) performs additional DH exchange in Phase 2 so compromise of one key does not expose past sessions.

Recommended for sensitive environments.

IPsec Tunnel vs IPsec VPN

These terms are often used interchangeably, although they commonly refer to different deployment models in operations.

TermPractical Meaning
IPsec TunnelSite-to-site routed or policy tunnel between devices
IPsec VPNGeneric term covering site-to-site or remote access secure VPN

In FortiGate context, site-to-site interface-based VPN is the most common enterprise design.

Example 1: Site-to-Site IPsec Tunnel (FortiGate <-> FortiGate)

Topology

Site-A LAN 10.1.0.0/24 --- FGT-A === Internet === FGT-B --- Site-B LAN 10.2.0.0/24

FortiGate A

config vpn ipsec phase1-interface
 edit "FGT-B"
  set interface "wan1"
  set remote-gw 198.51.100.2
  set peertype any
  set proposal aes256gcm-prfsha256
  set dhgrp 20
  set psksecret ENC myStrongKey
 next
end

config vpn ipsec phase2-interface
 edit "FGT-B-P2"
  set phase1name "FGT-B"
  set proposal aes256gcm
  set dhgrp 20
  set src-subnet 10.1.0.0 255.255.255.0
  set dst-subnet 10.2.0.0 255.255.255.0
 next
end

FortiGate B

config vpn ipsec phase1-interface
 edit "FGT-A"
  set interface "wan1"
  set remote-gw 203.0.113.2
  set peertype any
  set proposal aes256gcm-prfsha256
  set dhgrp 20
  set psksecret ENC myStrongKey
 next
end

config vpn ipsec phase2-interface
 edit "FGT-A-P2"
  set phase1name "FGT-A"
  set proposal aes256gcm
  set dhgrp 20
  set src-subnet 10.2.0.0 255.255.255.0
  set dst-subnet 10.1.0.0 255.255.255.0
 next
end

Required Policies

Create firewall policies allowing LAN <-> VPN interface traffic on both devices.

Example 2: IPsec VPN for Remote User (Dial-Up)

config vpn ipsec phase1-interface
 edit "RemoteUserVPN"
  set type dynamic
  set interface "wan1"
  set mode-cfg enable
  set proposal aes256-sha256
  set dhgrp 14
  set ipv4-start-ip 10.10.10.10
  set ipv4-end-ip 10.10.10.50
  set psksecret ENC RemoteUserKey
 next
end

This configuration enables remote users with compatible IPsec client software to establish secure connectivity.

Summary / Takeaways

  • IPsec secures Layer 3 traffic across untrusted networks.
  • FortiGate commonly uses IKE + ESP with Phase1 and Phase2 negotiation.
  • AES-GCM and ChaCha20 are strong modern choices.
  • DH groups define key exchange strength; Group 20 is an excellent enterprise option.
  • Site-to-site IPsec tunnels securely connect networks.
  • IPsec VPN can also serve remote users.
  • Strong cryptography must be paired with correct routing and firewall policy.

Effective VPN design combines strong cryptography, operational reliability, and predictable traffic flow.
