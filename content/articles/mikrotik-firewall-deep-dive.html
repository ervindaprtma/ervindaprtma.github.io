MikroTik Firewall Deep Dive: Filter, Mangle & Raw

  Ervinda Pratama   April 2025   ~10 min read

Introduction

When you start configuring a MikroTik router, you quickly encounter three firewall tables: Filter, Mangle, and Raw. Many engineers use only Filter by default, but understanding when and why to use each one is the difference between a functional firewall and a truly optimized one.

This article is based on my hands-on experience managing MikroTik-based ISP networks and security deployments, combined with concepts reinforced during MTCNA and MTCTCE certification studies.

"A firewall is only as good as the engineer who configures it." — Every NOC engineer, ever.

The Three Firewall Tables

TablePurposeWorks OnConnection Tracking
FilterAccept / Drop / Reject packetsProcessed packetsYes
MangleMark packets / routes / connectionsProcessed packetsYes
RawDrop before connection trackingAll packets (early)No

Filter Table

The Filter table is the primary tool for allowing or blocking traffic. It processes packets after connection tracking, which means you can make decisions based on connection state (new, established, related, invalid).

Common Chains

  • input — traffic destined for the router itself
  • forward — traffic passing through the router
  • output — traffic originated by the router

Example: Allow established, block the rest on Input

/ip firewall filter
add chain=input connection-state=established,related action=accept comment="Allow established"
add chain=input connection-state=invalid action=drop comment="Drop invalid"
add chain=input action=drop comment="Drop everything else on input"

Mangle Table

Mangle does not block or allow traffic — it marks it. These marks are used by other features like queues (QoS), routing policies (PBR), and ECMP.

Use Case: Mark HTTP/HTTPS for Separate Queue

/ip firewall mangle
add chain=forward protocol=tcp dst-port=80,443 action=mark-packet \
    new-packet-mark=web-traffic passthrough=yes

Then in a queue tree, you reference web-traffic to apply bandwidth limits independently.

Use Case: Policy-Based Routing (PBR)

/ip firewall mangle
add chain=prerouting src-address=192.168.10.0/24 action=mark-routing \
    new-routing-mark=isp2-route passthrough=no

Raw Table

Raw is the earliest interception point — packets hit Raw before connection tracking even starts. This makes it extremely efficient for dropping large volumes of attack traffic (DDoS, port scans) without ever loading the connection table.

Use Raw for known-bad source IPs or bogon prefixes. It protects CPU far more effectively than Filter rules.

Example: Drop Bogon Prefixes Early

/ip firewall raw
add chain=prerouting src-address-list=bogons action=drop comment="Drop bogon sources"

When to Use Which

ScenarioUse
Block unauthorized SSH access to the routerFilter — Input
Limit bandwidth for a specific subnetMangle (mark) + Queue
Route different traffic through different ISPsMangle (mark-routing) + Route
Drop DDoS flood traffic as early as possibleRaw
Block a forwarded traffic flow between VLANsFilter — Forward

Summary / Takeaways

  • Use Filter to make accept/drop decisions — it's connection-state aware.
  • Use Mangle to mark traffic for queues, PBR, or ECMP — never to block.
  • Use Raw to drop traffic at the earliest possible point, protecting CPU from connection tracking overhead.
  • Order matters: Raw → (connection tracking) → Mangle/Filter. Keep your rules specific and ordered for performance.
