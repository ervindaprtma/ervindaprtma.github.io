MikroTik QoS Deep Dive: Simple Queue, Queue Tree, Burst & HTB Design

  Ervinda Pratama   June 2026   ~20 min read

Introduction

Quality of Service (QoS) on MikroTik is one of the most powerful features available for bandwidth control, subscriber fairness, and traffic prioritization. In production environments, QoS is not only about limiting speed. It is about controlling congestion, preserving application performance, and distributing shared capacity intelligently.

MikroTik provides two major queue models:

  • Simple Queue
  • Queue Tree

Both rely on HTB (Hierarchical Token Bucket), the traffic shaping engine used to allocate bandwidth with parent-child hierarchy, guarantees, priorities, and burst behavior.

Why QoS Matters

Without QoS, bandwidth is usually consumed by the most aggressive traffic flows.

This often creates:

  • Voice jitter during downloads
  • Backup jobs saturating uplinks
  • One customer consuming shared ISP bandwidth
  • High latency during congestion
  • Unfair distribution among users

QoS transforms bandwidth from shared chaos into controlled service delivery.

Simple Queue Deep Dive

Simple Queue is the fastest method to apply bandwidth control to users, subnets, or IP ranges.

Example: Limit One Customer to 20 Mbps

/queue simple
add name="Cust-A" target=10.10.10.10 max-limit=20M/20M

Example: Branch Office Limit

/queue simple
add name="Branch-Users" target=192.168.10.0/24 max-limit=50M/50M

Advantages

  • Easy deployment
  • Fast troubleshooting
  • Suitable for SMB / WISP subscriber management

Limitations

  • Less granular than Queue Tree
  • Large scale with many entries needs structured design

Queue Tree Deep Dive

Queue Tree is designed for advanced QoS where traffic is classified first, then controlled under parent-child hierarchy.

Typical inputs:

  • Packet marks from Mangle
  • Interface parent
  • Parent queue hierarchy
  • Priority levels
  • CIR / MIR style allocation

Example Flow

Mangle marks VoIP packets -> queue tree high priority
Mangle marks Backup traffic -> queue tree low priority

Example Configuration

/queue tree
add name="WAN-TOTAL" parent=ether1 max-limit=200M
add name="VOICE" parent="WAN-TOTAL" packet-mark=voice priority=1 limit-at=20M max-limit=50M
add name="BACKUP" parent="WAN-TOTAL" packet-mark=backup priority=8 max-limit=80M

Understanding HTB (Hierarchical Token Bucket)

HTB is the queue scheduler used by MikroTik to shape traffic through token-bucket mathematics. Instead of sending packets without control, HTB allows packets to transmit only when enough tokens are available in the bucket.

A token represents permission to send a quantity of bytes. Tokens are continuously added to the bucket at a configured rate. When traffic consumes tokens faster than they refill, throughput is reduced to the defined shaping rate.

Core Token-Bucket Mechanics

Bucket fills over time -> tokens accumulate
Packets transmit -> tokens are consumed
Bucket empty -> traffic must wait for refill

This mechanism smooths bursts while still enforcing bandwidth policy.

MikroTik Bucket-Size Default = 0.1

In MikroTik queues, each rule includes a bucket-size parameter. The commonly used default value is 0.1.

Meaning of 0.1

Bucket-size is a multiplier of the configured max-limit (or available rate context) used to calculate temporary token storage.

Simplified model:

Bucket Capacity ≈ max-limit × bucket-size

So if max-limit is 100 Mbps:

100 Mbps × 0.1 = 10 Mb worth of token capacity

This means the queue may temporarily accumulate enough tokens to send short bursts before settling back to sustained shaping limits.

Traffic Impact of Default 0.1

ValueEffect
0.1Balanced default, moderate responsiveness
Lower (0.01)Tighter control, fewer bursts
Higher (1.0)Larger bursts, faster short transfers
Too HighPossible unfairness / micro-burst behavior

Why It Matters

  • Small bucket = smoother but stricter traffic control
  • Larger bucket = more responsive short downloads/uploads
  • Oversized bucket = traffic spikes may affect latency-sensitive flows

Calculation Example 1: Simple Queue

/queue simple
add name="User1" target=10.10.10.10 max-limit=20M/20M bucket-size=0.1/0.1

Approximate token storage per direction:

20 Mbps × 0.1 = 2 Mb token capacity

The user can momentarily transmit faster in short intervals until tokens normalize.

Calculation Example 2: Queue Tree Parent

/queue tree
add name="WAN" parent=ether1 max-limit=200M bucket-size=0.1

Approximate parent bucket:

200 Mbps × 0.1 = 20 Mb token capacity

Useful for absorbing short aggregate bursts across child queues.

ScenarioSuggested Bucket

Example: Low Latency Voice Queue

/queue tree
add name="VOICE" parent=WAN max-limit=20M priority=1 bucket-size=0.02

Small bucket reduces burst spikes and helps latency consistency.

Example: Fast Web Experience Queue

/queue simple
add name="Users" target=192.168.10.0/24 max-limit=100M/100M bucket-size=0.25/0.25

Larger bucket improves short web/object transfers.

Practical Tuning Steps

1. Start with default 0.1
2. Measure latency during congestion
3. Reduce bucket for voice / interactive traffic
4. Increase bucket for user browsing responsiveness
5. Re-test with real workloads

Important Notes

  • bucket-size is not guaranteed bandwidth.
  • It controls token storage behavior.
  • limit-at and max-limit still define rate policy.
  • Parent and child queues may interact during congestion.

Burst Limit Deep Dive

Burst allows temporary higher throughput when average traffic remains below threshold.

Useful for:

  • Fast web page loading
  • Short file transfers
  • Better user responsiveness
  • Controlled oversubscription environments

Example Queue with Burst

/queue simple
add name="User1" target=10.10.10.11 \
max-limit=10M/10M \
burst-limit=20M/20M \
burst-threshold=8M/8M \
burst-time=30s/30s

How It Works

If average usage over 30 seconds stays below 8 Mbps, the user may burst to 20 Mbps temporarily.

Once sustained usage rises above threshold, traffic returns toward normal max-limit policy behavior.

Provider Scenario (WISP / ISP)

Objective

Deliver fair shared internet to 500 subscribers on 1 Gbps uplink.

Recommended Design

  • Parent queue = 1 Gbps
  • PCQ child queues per subscriber tier
  • Burst for residential responsiveness
  • Priority for business customers

Example

Business Clients -> Higher priority + guaranteed CIR
Residential -> PCQ fairness
Night Backup -> Lower priority

Operational Benefit

  • Prevent one user saturating link
  • Better customer experience
  • Sell premium tiers efficiently

Data Center Scenario

Objective

Protect business applications during congestion.

Example Hierarchy

WAN/DC Edge 500 Mbps
 ├── Voice Gateway        priority 1
 ├── ERP / API           priority 2
 ├── Replication         priority 6
 └── Guest Internet      priority 8

Benefit

  • Replication never breaks voice
  • ERP remains responsive
  • Guest traffic uses excess bandwidth only

Queue Tree + Mangle Example

/ip firewall mangle
add chain=forward protocol=udp dst-port=5060 action=mark-packet new-packet-mark=voice passthrough=yes
add chain=forward dst-address=10.20.30.0/24 action=mark-packet new-packet-mark=erp passthrough=yes

Then attach packet marks to Queue Tree.

Common Misunderstandings

  • max-limit is not guaranteed bandwidth.
  • priority only matters during congestion.
  • Burst is temporary, not permanent speed boost.
  • Simple Queue is not always best at scale.
  • Queue Tree without good marking loses precision.

Summary / Takeaways

  • Simple Queue is ideal for straightforward user-based shaping.
  • Queue Tree provides enterprise-grade hierarchical QoS control.
  • HTB enables guarantees, borrowing, and priority scheduling.
  • Burst improves user responsiveness for short-duration traffic.
  • Providers use QoS for fairness and service tiers.
  • Data centers use QoS to protect business-critical applications.
  • Strong MikroTik engineers design queues based on traffic behavior, not only speed limits.
