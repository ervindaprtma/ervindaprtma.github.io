FortiGate Firewall Policy Deep Dive: Packet Flow, SSL Inspection & Security Profiles Design

  Ervinda Pratama   April 2026   ~28 min read

Introduction

FortiGate firewall policy is more than an allow-or-deny rulebase. It is the enforcement point where routing decisions, NAT, SSL inspection, identity controls, and multiple security profiles converge.

Modern enterprise traffic is heavily encrypted. As a result, understanding when to use certificate inspection versus deep inspection is critical for effective threat prevention.

This article explains FortiGate packet flow across internal, external, and published-service scenarios, followed by practical guidance for each major security profile.

Core Firewall Policy Components

A FortiGate policy commonly evaluates:

  • Incoming interface & Outgoing interface
  • Source address / identity
  • Destination address
  • Service / port
  • Schedule
  • NAT / VIP translation
  • SSL inspection profile
  • Security profiles
  • Logging action

Traffic must match policy criteria before advanced inspection profiles are applied.

Simplified Packet Flow Order

Ingress Interface
 -> Session Creation
 -> Route Lookup / VIP Match
 -> Firewall Policy Match
 -> NAT Decision
 -> SSL Inspection Stage
 -> Security Profile Inspection
 -> Forward / Deny
 -> Logging

Exact internal processing varies by firmware version, proxy/flow mode, and feature set.

Internal to Internal Traffic Flow

Example

User VLAN -> Application VLAN HTTPS/443

Typical Controls

  • East-west segmentation policy
  • Application Control
  • IPS
  • Antivirus (if file transfer relevant)
  • SSL inspection when required visibility exists

Best Practice

Do not treat internal traffic as trusted by default. Lateral movement prevention is a major security objective.

Internal to External Traffic Flow

Example

User LAN -> Internet SaaS / Web Browsing

Typical Controls

  • NAT enabled
  • Web Filter
  • DNS Filter
  • Application Control
  • IPS
  • SSL inspection depending privacy/risk model
  • Logging enabled

This is the most common place for layered security profiles.

External to Internal Traffic Flow

Example

Internet Client -> Public VIP -> Internal Web Server

Typical Controls

  • VIP / DNAT
  • IPS
  • Web Application Firewall (if applicable model/service)
  • Geo/IP restrictions
  • Rate limiting / shaping
  • Bot protection controls

Deep inspection is usually not applied to inbound public TLS unless FortiGate terminates TLS or acts as reverse proxy in the chosen design.

SSL Inspection Modes

Certificate Inspection (Standard SSL Inspection)

FortiGate validates TLS handshake metadata without decrypting payload content.

Visibility typically includes:

  • Certificate information
  • SNI / hostname visibility in many TLS flows
  • Basic reputation / domain decisions
  • Some policy matching context

Advantages

  • Lower privacy impact
  • Lower processing overhead than deep inspection
  • Easier deployment

Limitations

  • Cannot inspect encrypted payload malware/content deeply
  • Reduced visibility into applications using TLS payloads

Deep Inspection (Full SSL Inspection)

FortiGate performs TLS interception using a trusted CA certificate installed on endpoints.

Process:

Client trusts FortiGate CA
Client <-> FortiGate TLS session
FortiGate <-> Internet Server TLS session

Payload can then be scanned for threats and content controls.

Advantages

  • Full web/content visibility
  • Malware scanning in HTTPS downloads
  • Better application identification
  • Stronger DLP / URL control outcomes

Considerations

  • Requires certificate deployment to endpoints
  • Higher CPU/resource usage
  • Some certificate-pinned apps may fail
  • Privacy/legal review recommended

Major Security Profiles on FortiGate

Antivirus Profile

Scans web/file/mail transfers for malware signatures and heuristics.

Best Placement

  • User to Internet
  • File transfer zones
  • Remote access users

SSL Need

  • HTTP: no SSL issue
  • HTTPS: Deep inspection recommended for payload scanning

Example Custom

Block infected files
Quarantine suspicious archives
Enable archive scan within performance limits

IPS Profile

Detects exploit attempts, vulnerability probes, protocol abuse, command-and-control patterns.

Best Placement

  • WAN to DMZ
  • Internal to server zones
  • User to Internet

SSL Need

Without decryption, IPS visibility into encrypted payload is limited. Deep inspection improves coverage for HTTPS-borne exploits.

Example Custom

Critical/High signatures block
Medium signatures monitor initially
Server protection signatures on DMZ policy

Web Filter Profile

Controls access to URL categories, domains, safe search, content policy.

Best Placement

  • User browsing outbound traffic
  • Guest networks
  • Remote workforce policies

SSL Need

  • Certificate inspection often enough for category/domain enforcement using hostname visibility.
  • Deep inspection needed when full URL/path/content visibility is required.

Example Custom

Allow Business / Education
Warn Social Media
Block Malware / Phishing / Newly Observed Domains

DNS Filter Profile

Protects users at DNS resolution stage by blocking malicious or disallowed domain lookups.

Important Clarification

FortiGate does not need to become the client's DNS server in every deployment.

DNS Filter can operate when DNS traffic passes through FortiGate and the firewall can inspect DNS queries (UDP/TCP 53 or supported DNS-over-TLS/HTTPS controls depending features/version/design).

Common models:

Client -> External DNS through FortiGate transit path
Client -> Internal DNS resolver through FortiGate path
FortiGate itself as DNS service / resolver forwarder (optional design)

Best Practice

Using FortiGate as explicit DNS server is optional, not mandatory. The key requirement is that DNS queries traverse an inspectable policy path.

SSL Need

Traditional DNS: no SSL requirement. DoH/DoT control may require broader policy and TLS inspection strategies.

Example Custom

Block Malware Domains
Block Phishing Domains
Log Newly Registered Domains
Allow Business Exceptions List

Application Control Profile

Identifies applications regardless of port when detectable.

Best Placement

  • User internet access
  • WAN egress control
  • Shadow IT reduction

SSL Need

Deep inspection improves identification for many encrypted apps.

Example Custom

Block anonymizers / TOR
Throttle streaming media
Allow collaboration tools

AntiSpam / Email Security (Where Used)

Used in email gateway scenarios.

SSL Need

Depends on mail architecture (SMTPS/IMAPS etc.).

Prevents sensitive data exfiltration.

SSL Need

Best Practices

  • Apply heavier profiles only where justified by risk.
  • Separate server, user, guest, and admin policies.
  • Start monitor mode before blocking disruptive signatures.
  • Use certificate inspection as baseline when deep inspection is not feasible.
  • Deploy trusted CA carefully before enabling deep inspection broadly.
  • Review logs regularly and tune false positives.

Common Misunderstandings

  • Allow policy without profiles is not full protection.
  • Deep inspection is not required for every network segment.
  • DNS Filter does not always require FortiGate to be DNS server.
  • More profiles enabled does not always equal better security.
  • Internal traffic may need inspection as much as internet traffic.

Summary / Takeaways

  • Firewall policy is the control point where routing, NAT, SSL inspection, and threat prevention meet.
  • Certificate inspection offers lighter encrypted-traffic visibility; deep inspection offers full payload inspection.
  • Security profiles should be mapped to risk and traffic type, not enabled blindly everywhere.
  • DNS Filter works when DNS traffic is visible through FortiGate policy path.
  • East-west traffic inspection is increasingly important in enterprise environments.
  • Strong FortiGate design balances protection, privacy, performance, and operational manageability.
