KMS Secure File Archive: SoftHSM + PKCS#11 + Hybrid Encryption

  MasVin22 — Network Security Strategist   June 2026   ~18 min read   Implementation Note

Overview

KMS is a hybrid cryptography-based secure file encryption implementation that combines AES-256-GCM for data encryption and RSA-OAEP-SHA256 for key wrapping, with the private key fully managed by SoftHSM through the PKCS#11 interface.

The system consists of three primary components:

  • encrypt_test.py — file encryption, reading full configuration from .env
  • check_metadata.py — metadata inspection for .enc files without decryption and without HSM access
  • decrypt_test.py — file decryption, reading key identity directly from .enc metadata

The output of the encryption process is a KMS envelope file — a verified binary structure that stores all cryptographic materials required for decryption except for the private key itself.

Prerequisites and Configuration

Dependency Installation

.env File

The .env file is fully used by encrypt_test.py. decrypt_test.py only reads PKCS11_LIB. check_metadata.py only reads SECURITY_POLICY. 

PKCS11_LIB=/usr/lib64/softhsm/libsofthsm.so
PKCS11_TOKEN_LABEL=Test-Labels
PKCS11_USER_PIN=12341234
PKCS11_KEY_LABEL=RSA_WRAP_TEST
PKCS11_KEY_ID=00000002

# strict   = minimum RSA 3072-bit, hard exit if not compliant
# standard = minimum RSA 2048-bit, warning if < 3072-bit (DEFAULT)
# legacy   = minimum RSA 1024-bit, always displays warning
SECURITY_POLICY=standard

Section 1 — Generate RSA Key Pair

Tool: pkcs11-tool (OpenSC)
No Python script involved — this stage is purely HSM infrastructure preparation.

Token Initialization

If the token does not yet exist, initialize it first:

softhsm2-util --init-token \
  --slot 0 \
  --label "Test-Labels" \
  --pin 12341234 \
  --so-pin 00000000

--label must match PKCS11_TOKEN_LABEL, --pin must match PKCS11_USER_PIN, and --so-pin is the Security Officer PIN used for token administration.

Generate Key Pair

pkcs11-tool \
  --module /usr/lib64/softhsm/libsofthsm.so \
  --login --pin 12341234 \
  --token-label Test-Labels \
  --keypairgen --key-type rsa:2048 \
  --id 00000002 \
  --label RSA_WRAP_TEST

The never extractable attribute guarantees that the private key never leaves the HSM boundary in plaintext form.

Key ID Normalization

SoftHSM stores Key IDs with inconsistent leading-null padding. encrypt_test.py handles this through normalize_key_id():

def normalize_key_id(key_id_hex):
    binary_id = bytes.fromhex(key_id_hex)        # "00000002" -> b'\x00\x00\x00\x02'
    binary_id = binary_id.lstrip(b'\x00')         # -> b'\x02'
    if len(binary_id) < 4:
        binary_id = b'\x00' * (4 - len(binary_id)) + binary_id  # -> b'\x00\x00\x00\x02'
    return binary_id

During decryption, decrypt_test.py attempts three ID variations to accommodate vendor inconsistencies:

for search_id in [normalized_id, key_id_bin, b'\x00' + normalized_id]:
    objs = list(session.get_objects({...ID: search_id}))
    if objs:
        priv_key = objs[0]
        break

KMS Envelope Structure

The generated .enc file uses the following binary layout:

Offset   Size    Field           Description
------   ----    -----           -----------
0        4       Magic Prefix    b'KMS' (0x4B4D5332)
4        2       Version         0x0002 (METADATA_VERSION = 2)
6        2       Flags           0x0001 (authenticated encryption)
8        4       Timestamp       Unix timestamp (big-endian uint32)
12       4       Footer          0x53454346 = b'SECF'
[Header = 16 bytes total]

16       4       algo_len        Always 4
20       2       sym_algo        0x0001 = AES-256-GCM
22       2       asym_algo       0x0002 = RSA-OAEP-SHA256

24       4       key_id_len      Length of key ID string
28       N       key_id          "{KEY_LABEL}:{KEY_ID_HEX}" in UTF-8

28+N     12      IV              AES-GCM nonce 96-bit (fixed size)
40+N     4       wrapped_key_len Length of encrypted AES key (256 for RSA-2048)
44+N     256     wrapped_key     AES session key wrapped with RSA-OAEP-SHA256
300+N    16      GCM Tag         128-bit authentication tag (fixed size)
316+N    4       ciphertext_len  Ciphertext length
320+N    M       ciphertext      Encrypted file data

Secure Memory Handling

All sensitive materials are stored as mutable bytearray objects so they can be overwritten in-place. Immutable bytes objects are intentionally avoided for sensitive data. 

aes_key = bytearray(os.urandom(AES_KEY_SIZE))   # 32 bytes
data_iv = bytearray(os.urandom(IV_SIZE))         # 12 bytes

Wiping is performed inside a finally block using three-pass overwrite:

for pattern in [b'\x00', b'\xFF', os.urandom(1)]:
    for i in range(len(buffer)):
        buffer[i] = pattern[0]

Python does not guarantee full memory elimination due to garbage collection and internal string interning. This implementation is best-effort and not equivalent to Rust or C for complete memory safety.

Section 4 — File Decryption (decrypt_test.py)

Reads from .enc metadata: Key label and Key ID (not from .env)
Reads from .env: Only PKCS11_LIB as fallback library path
PIN: Entered interactively through getpass — never stored in environment variables

Usage

# Default output: backup.enc -> backup
python3 decrypt_test.py backup.enc

# Explicit output
python3 decrypt_test.py backup.enc output.pdf

# Decrypt + integrity verification against original plaintext hash
python3 decrypt_test.py backup.enc output.pdf \
  --expected-original-hash sha256_of_original_plaintext

Decoupling Key Identity from Configuration

decrypt_test.py does not read PKCS11_KEY_LABEL or PKCS11_KEY_ID from .env. The key identity is read directly from metadata inside the .enc file: 

metadata = parse_metadata(encrypted_file)
key_label, key_id_hex = metadata["key_id"].split(":", 1)
# Example: "RSA_WRAP_TEST:00000002"
# key_label  = "RSA_WRAP_TEST"
# key_id_hex = "00000002"

This is an intentional design decision. The .enc file carries its own key identity, allowing receivers to decrypt files without needing access to the sender's .env configuration.

OAEP Fallback — Technical Details

Some SoftHSM builds and PKCS#11 implementations reject specific OAEP parameters (SHA-256 + MGF1-SHA256). The system handles this through a two-stage fallback mechanism: 

Scenario Normal — Hardware OAEP:
  HSM:    RSA decrypt + OAEP unpadding  ->  AES session key
  Risk:   Minimal (all operations remain inside HSM boundary)

Scenario Fallback — Software OAEP:
  HSM:    RSA decrypt without padding (RSA_X_509)  ->  raw RSA block
  Python: OAEP unpadding in memory                 ->  AES session key

  Advantages:
  + Private key remains inside HSM
  + Constant-time comparison mitigates timing oracle attacks (Bleichenbacher-style)

  Risks:
  - Raw RSA block temporarily exposed in Python memory
  - Slightly larger attack surface compared to pure hardware unwrap

    +-> Wipe AES key + plaintext from memory

Security Notes

Existing Implementations

AspectMechanism
Key isolationPrivate RSA key marked never extractable inside HSM
Session keyNew AES key generated per encryption, never reused
Tamper detection128-bit GCM authentication tag — automatic exception if modified
Timing oracleConstant-time comparison during OAEP hash validation
Atomicitytempfile + os.replace() during encryption; "xb" flag during decryption
Brute force mitigationPIN retry limit (3 attempts) with exit before HSM lockout
Policy safetyInvalid SECURITY_POLICY = sys.exit(1), no silent fallback

Areas for Improvement

decrypt_test.py does not enforce RSA security policy checks — only encrypt_test.py and check_metadata.py currently implement them. Python cannot guarantee complete memory sanitization because of garbage collection and internal string interning; for high-assurance environments, consider C or Rust wrappers. The implementation currently lacks audit logging integration with syslog or SIEM platforms, key rotation workflows, and multi-user authorization (M-of-N).

SoftHSM is not a replacement for a physical HSM in production environments. It does not provide tamper resistance, hardware entropy generation, or compliance certification.

Summary / Takeaways

  • KMS uses hybrid encryption: AES-256-GCM for data encryption and RSA-OAEP-SHA256 for key wrapping, with the private key fully isolated inside SoftHSM through PKCS#11.
  • The .enc file is a KMS envelope — a self-contained binary structure storing IV, wrapped AES key, GCM tag, and ciphertext inside a single verified file.
  • check_metadata.py operates entirely without HSM access and without decryption — only metadata parsing and structural validation.
  • decrypt_test.py reads key identity (label and ID) directly from metadata inside the .enc file rather than .env, enabling decoupling between sender and receiver.
  • The OAEP fallback (RSA_X_509 + software unpadding) addresses PKCS#11 inconsistencies across certain vendors while using constant-time hash comparison to mitigate timing oracle attacks.
  • Security policies (strict / standard / legacy) are implemented as hard enforcement — invalid values immediately terminate execution without silent fallback.
  • SoftHSM is highly effective for validating PKCS#11 workflows in laboratory environments. For production deployments, migration to physical HSM platforms such as Thales Luna, Utimaco, or Nitrokey is necessary to achieve real tamper resistance and hardware-backed key custody.
