Building an IOC Threat Feed on FortiGate

  Ervinda Pratama   April 2025   ~5 min read   Research Note

Overview

An Indicator of Compromise (IOC) Threat Feed is a continuously updated list of known malicious entities — IP addresses, domains, and file hashes — published by threat intelligence providers. By integrating this feed into FortiGate, every policy lookup automatically checks against the latest threat data without any manual intervention.

This note covers three specific use cases implemented in a production environment:

  1. Blocking outbound connections from local hosts to malicious internet destinations (IPs & domains)
  2. Blocking file transfers containing known malicious file hashes
  3. Blocking inbound public access to a DMZ Nginx service from scanner IPs and known attackers

Why External Threat Feeds?

FortiGate's built-in AV and IPS databases are powerful, but they are signature-update dependent (pulled from Fortinet's cloud on a schedule). An External Threat Feed allows you to subscribe to any HTTP/HTTPS-hosted text list and have FortiGate consume it at a configurable interval — independently of Fortinet's update cycle.

This approach gives you the flexibility to use open-source IOC feeds (e.g., Abuse.ch, Emerging Threats, Feodo Tracker) or custom internal feeds generated from your own SIEM/SOC pipeline.

Use Case 1 — Block Local-to-Internet Traffic to Malicious Destinations

The goal here is preventive egress filtering: stop infected or compromised internal hosts from beaconing to C2 servers or downloading second-stage payloads.

Step 1: Create the External Connector

Navigate to: Security Fabric → External Connectors → Create New → Threat Feeds → IP Address

# FortiGate CLI equivalent:
config system external-resource
    edit "ioc-malicious-ip-feed"
        set type ip
        set resource "https://feodotracker.abuse.ch/downloads/ipblocklist.txt"
        set refresh-rate 60
        set status enable
    next
end

Repeat for a domain-type feed (e.g., Abuse.ch URLhaus):

config system external-resource
    edit "ioc-malicious-domain-feed"
        set type domain
        set resource "https://urlhaus.abuse.ch/downloads/hostfile/"
        set refresh-rate 60
        set status enable
    next
end

Step 2: Reference in Firewall Policy

Create or edit a policy covering the LAN-to-WAN direction. Add the threat feed object as a Destination with action DENY, placed above the general allow policy:

config firewall policy
    edit 0
        set name "Block-IOC-Egress"
        set srcintf "LAN"
        set dstintf "WAN"
        set srcaddr "all"
        set dstaddr "ioc-malicious-ip-feed" "ioc-malicious-domain-feed"
        set action deny
        set schedule "always"
        set logtraffic all
        set comments "Block outbound to IOC-listed IPs and domains"
    next
end
⚠️ Note: Domain-based blocking requires DNS inspection to be enabled on the policy so FortiGate can match the DNS query against the domain feed before the connection is established.

Use Case 2 — Block File Transfers Containing Known Malicious Hashes

FortiGate's FortiSandbox connector or a Threat Feed of type "Malware Hash" can be assigned to an Antivirus profile. When a file in transit matches a hash in the feed, the transfer is blocked immediately. 

config system external-resource
    edit "ioc-malware-hash-feed"
        set type malware-hash
        set resource "https://bazaar.abuse.ch/export/txt/sha256/recent/"
        set refresh-rate 120
        set status enable
    next
end

Assign this to an AV profile under Security Profiles → Antivirus → External Malware Block List, then attach the AV profile to the relevant policies.

Use Case 3 — Block Public Scanners from Reaching DMZ (Nginx)

Public-facing services (e.g., an Nginx reverse proxy in the DMZ) are constantly hammered by automated scanners, bots, and reconnaissance tools. Using a threat feed of known scanner IPs (e.g., Shodan scanners, MassScanner, GreyNoise noisy IPs) as a Source block on the WAN-to-DMZ policy provides automatic suppression of this noise. 

config system external-resource
    edit "ioc-public-scanners-feed"
        set type ip
        set resource "https://your-internal-feed-server/scanner-blocklist.txt"
        set refresh-rate 60
        set status enable
    next
end

config firewall policy
    edit 0
        set name "Block-Scanners-to-DMZ"
        set srcintf "WAN"
        set dstintf "DMZ"
        set srcaddr "ioc-public-scanners-feed"
        set dstaddr "nginx-dmz-vip"
        set action deny
        set schedule "always"
        set logtraffic all
        set comments "Block known scanners/attackers from reaching Nginx DMZ"
    next
end

IP addresses of previously detected attackers can also be appended to a custom feed file hosted internally — creating a self-learning blocklist driven by your own FortiGate logs or SIEM detections.

Monitoring & Verification

  • Check connector status: Security Fabric → External Connectors — verify "Last Sync" timestamp and entry count.
  • View blocked hits: Log & Report → Forward Traffic — filter by policy "Block-IOC-Egress".
  • CLI quick check:
# Check feed entries loaded
diagnose system external-resource status ioc-malicious-ip-feed

# Show last update time
diagnose debug rating

Key Takeaways

  • External Threat Feeds on FortiGate require no additional licenses — they work natively via HTTP/HTTPS pull.
  • Refresh rate of 60 minutes strikes a good balance between freshness and FortiGate API load.
  • Always place DENY+IOC policies above general allow policies in the policy table.
  • Domain feeds require DNS inspection; IP feeds work at the network layer — no inspection needed.
  • Combine external feeds with FortiGate's built-in botnet C2 list for layered coverage.
