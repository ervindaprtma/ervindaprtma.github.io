SD-WAN SLA Monitoring with Telegraf & FortiGate

  Ervinda Pratama   February 2025   ~6 min read   Research Note

Overview

FortiGate's SD-WAN feature continuously measures the health of each WAN link via SLA probes — tracking latency, jitter, and packet loss per link. While FortiGate's own GUI provides a basic SLA view, this is not persistent (no historical data) and cannot be integrated into a centralized dashboard.

This research note documents a pipeline that uses Telegraf (as an SNMP collector) to poll FortiGate SD-WAN SLA OIDs, writes the results to an OpenSearch index, and renders the time-series data in Grafana dashboards — giving operators a persistent, queryable view of WAN link health over time.

FortiGate SNMP & SD-WAN SLA OIDs

FortiGate exposes SD-WAN health-check (SLA) data through the Fortinet FortiGate MIB. The relevant OID table is fgVWLHealthCheck under the FORTINET-FORTIGATE-MIB.

Key OIDs

MetricOIDUnit
SLA Name1.3.6.1.4.1.12356.101.4.9.2.1.2String
Link Interface1.3.6.1.4.1.12356.101.4.9.2.1.3String
Latency (ms)1.3.6.1.4.1.12356.101.4.9.2.1.5Milliseconds
Jitter (ms)1.3.6.1.4.1.12356.101.4.9.2.1.6Milliseconds
Packet Loss (%)1.3.6.1.4.1.12356.101.4.9.2.1.7Percent
State (up/down)1.3.6.1.4.1.12356.101.4.9.2.1.41=up, 2=down
ℹ️ Note: These OIDs require the Fortinet MIB to be loaded in your SNMP manager. Download the MIB from FortiGate's official support portal or extract it from the firmware: /bin/mibs/ via SSH.

Enable SNMP on FortiGate

config system snmp community
    edit 1
        set name "telegraf-monitor"
        set status enable
        config hosts
            edit 1
                set ip 192.168.100.50/32    # Telegraf host IP
            next
        end
    next
end

config system snmp sysinfo
    set status enable
    set description "FortiGate-HQ-FW01"
    set contact-info "noc@company.net"
    set location "Jakarta-DC"
end

Step 1 — Telegraf Configuration

Install Telegraf on the monitoring server (Ubuntu):

wget -q https://repos.influxdata.com/influxdata-archive_compat.key -O- | \
    sudo gpg --dearmor -o /etc/apt/trusted.gpg.d/influxdata.gpg

echo "deb [signed-by=/etc/apt/trusted.gpg.d/influxdata.gpg] \
    https://repos.influxdata.com/ubuntu stable main" | \
    sudo tee /etc/apt/sources.list.d/influxdata.list

sudo apt update && sudo apt install -y telegraf

SNMP Input Plugin for SD-WAN SLA

Create /etc/telegraf/telegraf.d/fortigate-sdwan.conf:

[[inputs.snmp]]
  agents = ["udp://192.168.1.1:161"]    # FortiGate management IP
  version = 2
  community = "telegraf-monitor"
  interval = "60s"
  timeout = "10s"
  retries = 3

  ## SD-WAN SLA health check table
  [[inputs.snmp.table]]
    name = "fortigate_sdwan_sla"
    inherit_tags = ["source"]

    [[inputs.snmp.table.field]]
      name  = "sla_name"
      oid   = "1.3.6.1.4.1.12356.101.4.9.2.1.2"
      is_tag = true

    [[inputs.snmp.table.field]]
      name  = "link_interface"
      oid   = "1.3.6.1.4.1.12356.101.4.9.2.1.3"
      is_tag = true

    [[inputs.snmp.table.field]]
      name = "state"
      oid  = "1.3.6.1.4.1.12356.101.4.9.2.1.4"

    [[inputs.snmp.table.field]]
      name = "latency_ms"
      oid  = "1.3.6.1.4.1.12356.101.4.9.2.1.5"

    [[inputs.snmp.table.field]]
      name = "jitter_ms"
      oid  = "1.3.6.1.4.1.12356.101.4.9.2.1.6"

    [[inputs.snmp.table.field]]
      name = "packet_loss_pct"
      oid  = "1.3.6.1.4.1.12356.101.4.9.2.1.7"

Step 2 — OpenSearch Output Plugin

Add to /etc/telegraf/telegraf.d/output-opensearch.conf:

[[outputs.opensearch]]
  urls        = ["http://192.168.100.51:9200"]   # OpenSearch node
  index_name  = "telegraf-sdwan-%Y.%m.%d"
  manage_template = true
  template_name   = "telegraf-sdwan"
  username    = "admin"
  password    = "YourOpenSearchPassword"
  # Health check timeout
  timeout     = "5s"

Restart Telegraf and verify:

sudo systemctl restart telegraf
sudo journalctl -u telegraf -f

Step 3 — Verify Data in OpenSearch

# Check index created
curl -u admin:password http://192.168.100.51:9200/_cat/indices?v | grep sdwan

# Query latest documents
curl -u admin:password -X GET \
    "http://192.168.100.51:9200/telegraf-sdwan-*/_search?pretty&size=5" \
    -H 'Content-Type: application/json' \
    -d '{"sort": [{"@timestamp": {"order": "desc"}}]}'

Expected fields in each document:

{
  "@timestamp": "2025-02-15T10:00:00Z",
  "latency_ms": 12,
  "jitter_ms":   3,
  "packet_loss_pct": 0,
  "state": 1,
  "sla_name": "SLA-ISP1",
  "link_interface": "wan1",
  "host": "192.168.1.1"
}

Step 4 — Grafana Dashboard

Add OpenSearch Data Source

In Grafana: Configuration → Data Sources → Add → OpenSearch

  • URL: http://192.168.100.51:9200
  • Index pattern: telegraf-sdwan-*
  • Time field: @timestamp

⚠️ Note on Polling Interval: FortiGate updates SD-WAN SLA probe results approximately every 10–30 seconds depending on probe configuration. Setting Telegraf poll interval to 60s is safe and avoids overwhelming FortiGate's SNMP agent under high load. For higher resolution, reduce to 30s — but monitor FortiGate CPU impact.

Key Takeaways

  • FortiGate SD-WAN SLA data is accessible via SNMP using Fortinet's proprietary fgVWLHealthCheck OID table — no FortiAnalyzer license required.
  • Telegraf's SNMP input plugin natively supports table walks, making multi-link SD-WAN polling straightforward.
  • OpenSearch as the time-series backend offers powerful full-text query capabilities alongside time-series analytics — useful when correlating SLA degradation with security events in the same cluster.
  • Grafana dashboards provide historical trending that FortiGate's built-in GUI cannot — critical for SLA reporting and root-cause analysis.
  • The same Telegraf pipeline can be extended to collect FortiGate interface throughput, CPU, memory, and session count OIDs — creating a unified FortiGate observability dashboard.
